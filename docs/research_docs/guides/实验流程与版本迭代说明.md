# 实验流程与版本迭代说明（V1–V6）

> 历史说明：本文档描述的是旧版 V1–V6 迭代脉络。当前仓库正式路线请以 `docs/ROUTE_A_ARCHITECTURE.md` 与 `docs/EXPERIMENT_MASTER_GUIDE.md` 为准。

本文档用于在不依赖 zjuthesis 的前提下，统一说明本仓库的实验流程、核心模块职责、评测口径，以及从 V1 到 V6 的版本迭代脉络与对应产物位置。

## 2. 核心代码模块与职责

### 2.1 数据与测试集

- `benchmark_data.py`
  - `DOCUMENTS`：文档语料列表（doc_id 为索引）
  - `TEST_CASES`：测试用例列表，每条包含 `query` 与 `relevant_doc_ids`
  - V5阶段引入了“约束查询 / 无答案查询”类型，用于检验鲁棒性与拒答门控

### 2.2 RAG 流水线

- `rag_pipeline.py`
  - `_retrieve(query, top_k)`：阶段1，向量召回，返回带 `id/text/score/rank` 的候选列表
  - `_rerank(query, candidates)`：阶段2，重排序，返回带 `rerank_score` 的候选列表
  - `search(query, top_k)`：阶段3，全流程，可能触发 PRF 扩展与 RRF 融合
  - PRF + RRF：
    - 当置信度/阈值策略判定“需要扩展”时，构造扩展查询并检索/重排，再用 RRF 将两路结果融合
  - 无答案/拒答门控：
    - 当证据充分性不足（例如最高分低、分布不清晰）时，抑制扩展并给出更保守的结果

### 2.3 真实重排序器（V6）

- `real_reranker.py`
  - `RealReranker`：封装 `BAAI/bge-reranker-base` 的推理接口
  - `RerankerFactory`：在“真实/Mock”之间创建实例
  - 说明：若模型下载/离线环境不可用，流水线会回退到 legacy 或 mock 以保证实验可跑通

### 2.4 实验脚本与记录

- `reranker_study.py`（主基准实验入口）
  - `run_benchmark()`：逐条测试用例运行三阶段（检索→重排→全流程），统计 MRR/NDCG@5/P@3
  - `save_benchmark_result(...)`：将聚合指标写入 `research_history.json`
  - `generate_v2_report(...)`：生成中文 V5 报告到 `research_docs/reports/RERANKER_RESEARCH_REPORT_V2.md`

- `v6_experiment.py`（V6 对比实验）
  - 在同一测试集上对比 mock 与真实模型模式，输出对比结果 JSON，并写入 `research_history.json`
  - 如需允许在线下载模型，可通过环境变量控制（具体见脚本内说明）

## 3. 统一评测口径（每条 Query 的三阶段输出）

对每条 `TEST_CASES` 的 query，统一按如下顺序评测并累计：

1. 阶段1（向量检索）：`_retrieve(query, top_k=20)`
2. 阶段2（重排序）：`_rerank(query, retrieved_docs)`
3. 阶段3（全流程）：`search(query, top_k=10)`（可能触发 PRF+RRF 或拒答门控）

计算方式：

- `relevant_doc_ids` 与结果列表中 `id` 对齐
- 通过相关文档在结果列表中的 rank 计算：
  - MRR：第一个相关结果的倒数排名均值
  - NDCG@5：Top-5 排序质量
  - Precision@3：Top-3 精确性

## 4. 版本迭代说明（V1–V6）

本项目采用“逐步加难 + 逐步加机制”的迭代方式：先在可控小规模数据上跑通端到端，再逐步引入更贴近真实分布的困难样本，观察失败模式并引入对应的门控/融合策略。

### V1：基线跑通

- 目标：打通“向量检索 + 重排序 + 评测脚本”的最小可复现链路
- 重点产物：基线指标写入 `research_history.json`

### V2：歧义查询集

- 目标：暴露语义歧义导致的误检（多义词、近义混淆）
- 方法：引入歧义测试用例，观察重排序对干扰项的惩罚能力
- 关键现象：向量检索容易“语义近但意图不符”，重排序可提升 Top-$k$ 精排稳定性

### V3：否定/约束查询

- 目标：检验系统在“NOT/EXCEPT”等约束条件下的鲁棒性
- 方法：引入带否定或排除条件的测试用例，观察 PRF 是否会放大噪声
- 关键现象：PRF 在低置信场景可能带来语义漂移，需要门控策略

### V4：扩展性与宽泛主题

- 目标：在文档规模增加、主题更宽泛时检验指标与稳定性
- 方法：扩展语料与查询类型，观察三阶段指标的变化趋势

### V5：无答案安全 + 自适应门控

- 目标：在“无答案”场景下避免高置信误检，降低生成端幻觉风险
- 方法：加入无答案测试用例；引入拒答门控与更谨慎的 PRF 触发策略
- 产物：
  - `research_docs/reports/RERANKER_RESEARCH_REPORT_V2.md`（运行脚本更新）
  - `research_history.json`（历史档案）

### V6：真实模型接入与对比验证

- 目标：从 Mock 评分过渡到真实 reranker 推理；验证 V1–V5 结论在真实推理下是否仍成立
- 方法：同一测试集并行跑 mock/real 两套，并输出对比报告
- 产物：
  - `research_docs/plans/V6_ITERATION_PLAN.md`
  - `v6_experiment.py` 的结果 JSON 与 `research_history.json` 的迭代追加

## 5. 最常用运行路径（建议）

如果只跑“主基准”并更新报告/历史：

- 运行 `reranker_study.py`（生成报告 + 写入历史）

如果要验证 V6 mock vs real 对比：

- 运行 `v6_experiment.py`（输出对比 JSON + 写入历史）

## 6. 如何阅读 research_history.json

该文件以“迭代”为单位记录：

- `id`：迭代编号
- `timestamp`：运行时间
- `description`：本轮改动摘要
- `metrics`：聚合指标（通常包含检索与最终两个阶段的核心指标）

通过这种记录方式，可以用同一套查询集在不同版本间做横向对比，并把“改动—指标变化”关联起来，支撑论文写作时的“方法—实验—结论”叙事。

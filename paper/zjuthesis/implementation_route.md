# 开题实施路线（可落地版）

本文路线以“效果—成本—可信性”三维度为约束，按“证据入口 → 检索控制 → 推理链 → 验证闭环”递进推进。整体策略是先把数据、索引、评测闭环做稳，再逐步引入推理链与验证机制，避免一开始就承诺高风险且难以验收的“统一跨模态嵌入训练”。

## 0. 目标与边界

本课题的目标是完成一个面向多源异构知识的RAG原型系统：在多跳任务上相对Baseline取得可量化提升，同时能够输出可追溯证据与可回放推理路径，并在实验报告中给出可复现的对比与消融结论。为保证可落地性，系统边界收敛为“统一证据入口 + 来源追溯 + 基础实体对齐（别名归一/ID映射）”，将版本/时序一致性与跨载体强互证视为可选扩展，不作为首要验收项。评测时除效果指标外，必须同时记录端到端延迟、LLM调用次数与Token消耗，作为与效果并列的硬约束。

## 1. 子任务与递进关系

### 子任务1：统一证据入口与可追溯定位

子任务1的核心是把“文本/表格/KG”统一抽象为证据单元，并且每条证据都能精确定位到来源位置以便追溯与展示。数据侧建议以三类输入落地：文本（Wikipedia/论文PDF转文本或企业文档）、表格（CSV或从PDF抽取的表格）、局部KG（三元组）。证据单元统一为三种形态：文本chunk、表格row、KG triple，并统一附带定位信息（来源文件、页码/行号、实体ID或标题）。系统实现上，文本侧采用切分与去重后构建向量索引（FAISS）与倒排索引（Elasticsearch）；表格侧以“行”为基本证据单元，将字段—值序列化后进行嵌入以便向量检索；图谱侧以三元组（head, relation, tail）作为证据单元，存入Neo4j并支持路径/邻居查询。该子任务的验收标准是：任一检索结果都能回溯到原始位置（文件名、页码/行号或三元组来源），并能在展示层输出Top-$k$证据的标题、片段与来源。

### 子任务2：成本可控的自适应检索与证据选择

子任务2的核心是把检索从“固定一次召回”升级为“策略可控的按需检索”，并明确把时间与调用成本纳入优化目标。输入为问题$q$、可为空的推理状态，以及子任务1构建的证据索引；输出为检索策略（来源选择、检索算子、Top-$k$、是否二次检索与预算）与证据集合。落地上先实现Query Router：用规则或轻量分类把问题路由为文本型、表格型、需要KG或混合型，并据此决定优先检索来源；然后实现触发策略：限制最大检索轮次（建议不超过2），并对Token与时间做预算上限；最后实现证据选择：在候选证据上用交叉编码器重排，必要时用RRF做排序融合。验收标准是：在不显著降低效果指标的情况下，平均LLM调用次数与端到端延迟下降；同时系统保存完整的路由与触发日志，能够回放每次检索为什么触发、选了哪些证据以及每步成本。

### 子任务3：证据链/推理链构造的复杂推理机制

子任务3的核心是把“多段证据”组织成可回放的推理结构，使多跳问题从“在段落集合里搜索”转化为“沿链条推进”。输入为证据集合（chunk/row/triple）与问题$q$，输出为推理链（或推理图）与最终答案。建议采用先易后难的两阶段路线：第一阶段先做文本多跳链式推理，流程为问题分解、每跳检索、合并证据并回答，以确保基本可用；第二阶段再引入TRACE式推理链思想，把检索文档抽取为知识三元组并自回归选择形成推理链，再以推理链作为高信噪比上下文生成答案，从而提升鲁棒性与解释性。验收标准是：系统能够输出推理链（每一步对应具体证据单元并带来源定位），并且在多跳数据集上相对Baseline取得可量化提升（EM/F1）。

### 子任务4：可验证闭环与系统级评测

子任务4的核心是把推理过程闭环为“可验证、可拒答、可归因”的系统行为，降低证据不足时的高置信错误输出风险。输入为候选答案与证据链/推理链，输出为支撑性判定（支持/部分支持/不支持）、冲突提示与拒答/回退决策。实现上先做支撑性判定：用规则或轻量提示对答案是否被证据支撑做分类；再做冲突检测：对同一实体属性出现多值、互斥事实或来源矛盾时触发二次检索或输出保守结论；最后统一结果输出格式为“答案 + 证据列表 + 推理链 + 置信信号”。验收标准是：证据支撑率、冲突触发率等可信指标可统计；失败案例可复现并能定位到模块级原因（检索、重排、链条选择或验证策略）。

## 2. 数据集与评测口径

评测数据集分两组推进。纯文本多跳用于验证复杂推理能力，建议使用HotpotQA、2WikiMultiHopQA与MuSiQue；表格/文本混合用于验证异构融合是否真的用上结构化证据，建议使用HybridQA、OTT-QA与WikiTableQuestions。指标口径需要同时覆盖效果、成本与可信性：效果侧报告EM/F1以及检索相关指标（MRR、NDCG@k、Precision@k）；成本侧记录端到端延迟、LLM调用次数与Token消耗；可信性侧统计证据支撑率与冲突触发率，拒答正确率可作为扩展指标。

## 3. 里程碑与交付物（对应进度表）

里程碑按“可复现交付”组织。阶段1完成文献调研、问题定义与评测口径固化，交付综述与口径文档、评测脚本骨架与基础环境配置；阶段2完成证据入口与Baseline，交付索引构建脚本、Baseline流水线与基线指标报告；阶段3完成自适应检索与推理链原型，交付检索控制器、推理链模块以及关键消融结果；阶段4完成验证闭环与系统集成，交付可回放证据链、验证/拒答策略、Demo界面与完整评测报告；阶段5完成论文与答辩材料，交付论文定稿、答辩PPT与复现材料。

## 4. 风险与降级方案

本路线在风险控制上遵循“先可用、再提升”。如果跨载体强对齐难以稳定实现，则以基础实体对齐与来源追溯保证可审计性，把高级对齐与版本约束降级为扩展；如果多轮检索导致耗时过高，则严格限制最大轮次与预算，优先使用并行、缓存与重排融合，并用成本指标约束策略；如果推理链方法不稳定，则先用问题分解的链式多跳保证可用，再引入TRACE式链条提升解释性与抗噪声能力。

## 5. 更细步骤（数据、Baseline、系统构建与部署）

在具体实施上，建议把工作按“数据准备—Baseline建立—模块迭代—部署与复现”串成一条可以重复执行的流水线。首先在数据准备阶段，按两条线并行：纯文本多跳线下载并整理HotpotQA、2WikiMultiHopQA与MuSiQue的开发集/训练集样例，统一转成“问题—候选文档—标准答案—支持证据（若有）”格式；异构线准备HybridQA、OTT-QA与WikiTableQuestions，并将表格证据抽象成行级证据单元（行号、字段名、值）并与问题关联。随后在证据入口阶段，文本按切分策略生成chunk并写入FAISS与Elasticsearch；表格按行序列化生成向量并写入向量索引；若引入KG，则在小规模样例上先做局部三元组抽取并写入Neo4j，确保每条三元组能回溯到来源文本/表格位置。

在Baseline建立阶段，建议至少给出三条对照：第一条是BM25或Elasticsearch检索 + 直接拼接Top-$k$到LLM生成；第二条是向量检索（DPR/通用embedding）+ 直接拼接Top-$k$生成；第三条是向量检索 + 交叉编码器重排（rerank）+ 拼接Top-$k$生成，并可加入RRF作为融合对照。Baseline评测时固定同一生成提示词模板，统一输出EM/F1与检索指标，并记录每个样本的端到端延迟、调用次数与Token消耗，形成可复现的基线报告。

在模块迭代阶段，先加入Query Router与触发策略把检索控制从“固定一次”升级为“按需”，并通过日志验证每次路由与触发决策可解释；再在多跳数据集上加入推理链：先用问题分解的链式多跳实现可用版本，再引入TRACE式三元组推理链以减少噪声与压缩上下文，最后接入支撑性判定与冲突触发形成验证闭环。每次迭代都要求跑同一套评测脚本，输出“效果—成本—可信性”三维报告，并配套5–10个失败案例回放，用于定位改进方向。

在部署与复现阶段，建议把系统拆成三类可运行组件：索引构建组件（离线，输入数据输出索引文件与统计）、在线检索与推理服务组件（对外暴露HTTP接口或CLI）、以及评测组件（固定脚本与配置文件）。部署形态可以先以本地脚本与配置文件固化，再封装为服务以支持Demo展示；重点是保证配置可追溯、结果可复现、日志可回放，确保答辩时能展示从“问题—证据—推理—验证—答案”的全链路过程。
